实际上，我们并非完全“从零开始”，概念是我们的武器。

一个塔防游戏需要什么？

我们首先考虑战斗界面，因为它是游戏的主体与核心，是塔防游戏不可或缺的部分。

而其他界面是次要的，他们完全可以留到后面再来完成（咕咕咕……）
那么问题简化成：

一个战斗界面需要什么？

从全局考虑，战斗发生在一张地图上，并有敌我双方：

  1. 怪物从地图上的出怪口出发，并沿着既定路线前进，直到到达我方的基地为止。
    敌方的战略目的是将尽可能多的敌方单位送至我方基地中（给博士手撕）

  2. 我方可以在地图上的特定位置部署单位，阻拦或攻击敌人。
    我方的战略目的是尽可能地阻止敌人接近我方基地。

那么逻辑结构便明确了，一个地图类、一个我方单位类、一个敌方单位类。

最后我再用一个 GameState 类管理全局游戏状态，大功告成。

完成它们就够了吗？

当然不够，我们还需要摩多摩多~

但在这之前，我们必须认识到：要让程序运转起来的最小代码量是什么。这就是程序的核心框架、主要矛盾。

《口技》虽繁，盖“一人、一桌、一椅、一扇、一抚尺而已”，程序设计亦是如此。

动起来！

基本数据结构的设计，我们在上一节已经大致完成。虽然它暂时还略显简陋，不过就其功用来讲已是绰绰有余。

下面的问题是，如何让代码动起来？

毕竟俗话说得好：

代码和人，总得有一个能跑的。

如何动？

Qt 提供了简单的 QTimer 类来帮助我们完成这个需求。

QTimer* _timer = new QTimer(this) // 创建一个计时器
_timer->start(20); //* 每 20ms 更新一次！
connect(_timer, &QTimer::timeout, this, &GameState::update); // 每次更新时调用 update 函数

size_t _time = 0; // 记录从计时器开始计时至今过去了多少帧。

发生在连续量上的问题，考虑起来总是很复杂的。

但如果我们能把连续量离散化，分别考虑每个离散点，结果说不定就会不一样。正如我们微积分时所做的那样。

QTimer 类的对象启动后，每隔一定时间触发一次更新，因此我们只需要编写自己的 update 函数就好啦~

只要相邻两次更新的间隔足够短，在玩家看起来游戏就是连续的。

void GameState::update()
{
    _time++;
    try {
        reunionStragegy(); // 敌方出怪策略
        reunionAction(); // 所有敌方依次行动
        operatorAction(); // 所有我方依次行动
    } catch (GameOverException& exception) {
        gameOver();
    }
    stateInfoUpdate();
}

带有函数式编程风格的代码，非常简洁漂亮。

书接上文，这里之所以采用 GameState::update 统一调配各单位的行动，而非各单位分别计时、自己安排自己的行动，一是为了逻辑上的简洁，二是为了避免程序中多线程导致的数据竞争。

试考虑以下情景：

我方单位 A 和 B 同时索敌敌方单位 C，A 先手击杀单位 C，但 C 已经在 B 的索敌状态下，此时有两个选择：

1. 析构 C，则出现悬空指针，B 攻击时报错。

2. 不析构 C，此时 B 只能再次攻击“死亡”后的 C，而这显然是荒谬的。

综上所述，多线程常常会导致奇怪的不安全性，因此除非有足够充分的理由，需要尽量避免使用。

但是行动很慢……

我们很快就会发现上文的不足之处：每 20ms 更新一次游戏状态，但是对象的行动，如移动或攻击频率却肯定不是 20ms 一次的，不然会让人眼花缭乱的……

怎么办？

一个简单的解决方案是给每个单位设定一个行动间隔，记录其两次行动间需要隔开的帧数。

每次行动后，只有至少又过去了这么多帧，才进行下一次行动，否则等待。

举个栗子，我把敌方战斗兵种“无人机”的 action 函数这么编写：

void UAV::action(size_t time, size_t& hp, Infected* op, Map* map)
{
    // 攻击和移动共享行动间隔
    if (time - _last_action_time < _interval) {
        return;
    }
    // 重置行动冷却时间
    _last_action_time = time;

    // 如果攻击范围内有敌人，则攻击！
    if (op != nullptr) {
        attack(op);
    }
    // 无论是否攻击，移动总是正常进行
    move(hp, map);
}

我们会发现，在上述代码编写下的无人机，采取的是畏战策略，即“边打边跑”。稍微变动两行代码，我们就可以得到好战类型的兵种：

void GroundReunion::action(size_t time, size_t& hp, Infected* op, Map* map)
{
    if (time - _last_action_time < _interval) {
        return;
    }
    _last_action_time = time;

    // 只要攻击范围内存在敌人，就会站着不动、持续进行攻击
    if (op != nullptr) {
        attack(op);
    } else {
        move(hp, map);
    }
}

我方单位的行动策略相对来说更简单一些，因为只用攻击、不用移动。

从现在开始迭代

迭代的过程和滚雪球很像，不过这可是门精细得多的手艺。

demo 的展现形式是命令行，可最终我们要的呈现效果可是图形化界面，要做好这个转换，最重要的地方是移动的连续化。

移动的连续化

所谓移动的连续化，就是每帧都进行一次移动。只有这样，我们看到的动画才不会是一个怪物突然就从一个格子“瞬移”到另外一个格子，而是逐渐地移动到标定位置。

虽然逻辑层中怪物的位置仍然与地图格子相绑定，但这不妨碍我们在图形层中给它一个平滑的过渡~

假设我们需要怪物 A 在 t 帧内横向移动一格，每格宽度为 100 个像素点，那么它每帧移动的像素点数 _move_speed = 100 / t。我们把这个结论拓展到二维空间中，引入二维速度向量，即可达成坐标的有效更新：

void Reunion::move(size_t& hp)
{
    // 按速度向量更新坐标
    setGeometry(x() + direction.first, y() + direction.second, 90, 90);

    // 判断是否到达下一个路径点，误差一个像素点
    if (abs(x() - (_route.back()->x() + 10)) <= 1 && abs(y() - _route.back()->y()) <= 1) {

        // 在逻辑层上，移动至下一个路径点
        removeFrom();
        addTo(_route.back());
        _route.pop_back();

        // 更新速度向量
        direction.first = (_route.back()->x() - _place->x()) / 100 * _move_speed;
        direction.second = (_route.back()->y() - _place->y()) / 100 * _move_speed;

        // 如果到达我方基地，移除该怪物，并减少基地生命值
        if (_place->isBase()) {
            _is_active = false;
            if (--hp == 0) {
                throw LoseGameException();
            }
        }
    }
}

看似大功告成，实际上这段代码并不是完美的：

在某些前提条件，如特殊的 _move_speed 值或斜飞路径下，速度向量的各分量可能并非整数。

但 (x,y) 坐标点必须为整数，那么速度向量的小数部分就会被舍弃。

每 20ms 进行移动时，新坐标都会向着固定方向产生固定偏差，则最后的误差可能是难以想象的，甚至会导致无法到达预计目标点。

这个问题几乎折磨了我一夜，最后的解决方案如下：

分别记录速度向量的小数部分和整数部分，整数部分每帧直接加到坐标上。

小数部分每帧独立计算累加和，一旦累加和向量某分量的绝对值大于 1，就在当前帧把 ±1 加到坐标对应分量上，±1 的符号取决于溢出分量的符号。

事实上，这个解决方案十分精确，在斜飞时，它可以误差两个像素点到达指定位置。

但源代码过长，就不放上来了，感兴趣的读者可至我的 github 仓库内下载查看。

干员朝向机制

如果让我说《明日方舟》区别于其他塔防游戏的独特之处，我想阻挡数和干员朝向是当之无愧的两颗明珠。

有限的阻挡数使得有选择地漏怪成为可能；
而干员朝向特化了我方单位的攻击范围，极大地丰富了游戏部署策略的可能。

阻挡数的实现较为简单，我下面主要说一说干员朝向机制：

加入干员朝向机制后，我方单位的实际攻击范围由三个元素共同决定：

干员纸面攻击范围、干员部署位置、干员朝向。

  1. 干员纸面攻击范围是干员的属性，一般来讲是一个矩形。我这里用 QPair<int, int> getAttackArea() 来简单表示。
  2. 干员部署位置由玩家在游戏时确定，是地图上的一个点。由于在逻辑层中，我们之前已经把地图离散化表示为二维地图格数组，这里可以直接用一个格子来表示干员部署位置。
  3. 干员朝向也由玩家指定，且必然是 上、下、左、右 中的一个方向，我们完全可以对其进行分类讨论。

最后，干员的实际攻击范围也可以表示为格子的集合。由于格子与其上的怪物也是绑定的，这实际上有利于我们的索敌机制。

由于 GameState 全知全能，我们让它来在部署干员时计算其有效攻击范围：

void GameState::findAttackPlaces(Operator* op) const
{
    int x = 0, y = 0, height = 0, width = 0;
    // 获取干员朝向，并参考其纸面攻击范围，确定相对于部署位置的实际攻击范围。
    switch (op->getOrientation()) {
    case UP:
        height = op->getAttackArea().second;
        width = op->getAttackArea().first;
        x = op->getPlace()->showID().second - width / 2;
        y = op->getPlace()->showID().first - (height - 1);
        break;
    // 其余三个方向同理，这里不再赘述
    }

    // 在表示地图的二维数组中，取出所有被该干员攻击范围覆盖的格子，注意不要访问越界。
    QVector<Place*> attack_area;
    for (int i = max(x, 0); i < min<int>(x + width, _map->getWidth()); i++) {
        for (int j = max(y, 0); j < min<int>(y + height, _map->getHeight()); j++) {
            attack_area.push_back((*_map)[j][i]);
        }
    }

    // 把攻击范围传递给干员保存。
    op->setAttackPlaces(attack_area);
}

template <class T>
inline T min(T x, T y)
{
    return (x < y) ? x : y;
}

template <class T>
T getRandomUnit(QVector<T>& units)
{
    srand(static_cast<unsigned>(clock()));
    return units[rand() % units.size()];
}

优化交互方式

既然有了图形化界面了，对应的鼠标交互也得跟上才是~

在本游戏中，交互的大头落在了部署干员这件事上。那么又回到那个熟悉的问题：

把大象放进冰箱需要几步？

对应的，我们必须把“干员部署”这一概念逐层细分，类似于广度优先搜索那样，才能知道我们应该如何做。

把干员放进战场需要几步？

第一步：选择要部署的干员

第二步：选择干员部署位置

第三步：选择干员朝向

我计划这三步都由鼠标完成互动，那么我们必须在后面的步骤中记录下前面步骤的状态：

  1. 第一步中：记录下所选择的干员，直到第三步结束
  2. 第二步中：记录下干员部署位置，直到第三步结束

第一步比较好解决：我把干员的大头照以 QLabel 形式放在游戏区域右下角，每次点击的时候，用 _operatorSelected 记录下所点击的干员。

第二、三步，《明日方舟》中采取的是分别进行的模式。我做了一个小小的创新：

点击要部署的格子并向 上、下、左、右 中的一个方向拖动光标，即可同时完成二、三步。

这一点在实现上还是分为两部分的，且听我细细道来~

伟大的 Qt 提供了鼠标点击事件和鼠标释放事件的支持，由于干员是与格子而非地图绑定的，所以我对 Place 类的对应函数进行重载：

void Place::mousePressEvent(QMouseEvent* event)
{
    // 当鼠标左键按下时，记录下点击处的坐标
    if (event->button() == Qt::LeftButton) {
        _prees_point = event->pos();
    }
}

void Place::mouseReleaseEvent(QMouseEvent* event)
{
    // 仍然是当鼠标左键释放时才触发
    if (event->button() != Qt::LeftButton) {
        return;
    }

    // 以释放处坐标与点击处坐标相减，得到鼠标位移向量 delta
    int delta_x = event->x() - _prees_point.x();
    int delta_y = event->y() - _prees_point.y();

    // 判断鼠标总的来说，是向左右位移得多还是向上下位移得多
    // 然后进一步，由位移的正负性判断具体的位移方向
    // 最后调用 GameState 的友元函数 deployOperator，把信息都交给它，完成干员的部署。
    if (abs(delta_x) > abs(delta_y)) {
        if (delta_x > 0) {
            deployOperator(_gamestate, this, RIGHT);
        } else {
            deployOperator(_gamestate, this, LEFT);
        }
    }
    if (abs(delta_x) < abs(delta_y)) {
        if (delta_y > 0) {
            deployOperator(_gamestate, this, UP);
        } else {
            deployOperator(_gamestate, this, DOWN);
        }
    }
}

这里要注意一点：必须监测鼠标的点击和释放事件，而不能检测鼠标移动事件。

如果我们把鼠标的整个移动过程当作一个整体，那么只有点击位置和释放位置是能代表这个整体的。

每一帧的移动都只是局部，而局部的移动方向往往不能充分反映整体，望周知。

这其实与平均速度、瞬时速度之间的关系有异曲同工之处。

此外，让我们回顾一下刚才的整个过程：

先分步处理，每一步都造成状态的改变，最后应用这些状态的累积，得到目标计算结果（一个部署好的干员）。

不知道各位读者有联想到什么没有，在这一节，我们其实实现了一个微型的有限状态机，或者不如说：

我们的整个程序都只不过是一个稍大点的有限状态机。

程序设计，很奇妙吧~

让小人动起来

如何让场上的单位都有自己的待机动画呢，这实际上与先前的 update 函数更新有着相同的原理：

只要相邻两帧之间的间隔足够短，相邻图片的内容差异足够小，图片的连续播放就会看起来像动画一样。

Qt 为我们提供了 void paintEvent(QPaintEvent*) 虚函数，每当我们调用同一单位的 update 函数时，paintEvent 函数就会紧接着被调用。

所以我们只需要一系列的图片，然后把它们扔进 paintEvent 函数里，就能做出人物的待机动画？

NO NO NO ~

事实上，伟大的明日方舟 wiki （https://prts.wiki ）已经为我们准备好了所有人物的 gif 动画。我们只需要每隔 20ms 从 gif 里截取一帧，设为人物的当前画面，就可以啦！

快说：谢谢明日方舟 wiki。

并且，为了人物动画看起来不那么单一，我们在攻击时设置 _is_attacking 标志位，绘出人物图像时按照 _is_attacking 的真值，选择截取不同的 gif：

void Operator::paintEvent(QPaintEvent*)
{
    QPainter painter(this);
    QImage image;
    // 人物总是在攻击状态和待机状态之间切换，选择截取需要的动画，将不需要的暂停，以节约资源
    if (_is_attacking) {
        _attack_movie->start();
        _idle_movie->stop();
        image = _attack_movie->currentImage();
    } else {
        _idle_movie->start();
        _attack_movie->stop();
        image = _idle_movie->currentImage();
    }
    painter.drawImage(QRect(-50, -70, 200, 200), image);
}

这样简单的播放暂停会导致另一个问题，即攻击与攻击动画不同步。

我暂时还没有想到什么简单明了的解决方案。

可是就算这样，怎么还是显得比较单调捏……

所以我们再增加一个受击动画，让受击单位泛红闪烁一下。为了过渡的自然性，再为这个泛红程度增加一点渐入渐出的效果。

让我们分解这个问题：

  1. 首先是受击的时候，要设置一个倒计时 _is_attacked_count = 10。每 20ms 更新一次，则泛红状态持续 200ms。
  2. 然后在 paintEvent 的重载中调用 convertToRedImage 函数，处理所需要被展示的图片。
  3. 在 convertToRedImage 内部，我们希望在 200ms 内，泛红的程度的增加量能先线性地增加，再线性地降低。
    于是求出 R 分量增加量函数 100 - 20 * abs(_is_attacked_count - 5)，并将其写入 convertToRedImage 中。

作函数如下：

void Infected::convertToRedImage(QImage& img, int degree)
{
    //* 四通道 8bit-8bit-8bit-8bit 的 rgba 色彩模式，前三位分别表示 rgb，最后一位表示透明度
    //* 8bit 为每位的值域，每个像素点的四位在色彩空间数组中临近存储
    img = QImage(img).convertToFormat(QImage::Format_RGBA8888);
    uint8_t* rgb = img.bits();

    //* 将 R 分量二次函数式地增加 300 个点，然后按同样规则衰退回原状态
    for (int i = 0; i < img.width() * img.height(); i++) {
        int new_red_degree = rgb[i * 4] + (100 - 20 * abs(degree - 5));
        rgb[i * 4] = new_red_degree > 255 ? 255 : new_red_degree;
    }
}

随后在刚才的 paintEvent 中 here 的位置添加两行代码即告成功：

if (_is_attacked_count > 0) {
    convertToRedImage(image, _is_attacked_count);
    _is_attacked_count--;
}

正如前文已经提到过的，干员攻击范围同时受干员纸面攻击范围、干员部署位置、干员朝向三个属性影响。

对于这种复杂的、不直观的内容，用易于理解的、直观的形式表示，这正是图形化界面的初衷。

来吧，还记得我们的干员攻击范围在逻辑层是用什么存储的吗？地块的集合。攻击范围的显示也是在地块上。

这就是说，对于一个干员，我们可以简单地召集他攻击范围内的所有地块，让它们分别显示出蓝色条条，就完成了整个攻击范围的显示。这正是：

为了保证只要鼠标处于点击状态，就显示攻击范围，所以干员类的鼠标点击函数和鼠标释放函数都应该重载：

void Operator::mousePressEvent(QMouseEvent* event)
{
    if (event->button() == Qt::LeftButton) {
        for (Place*& place : _attack_places) {
            place->paintAttackArea();
        }
    }
}

void Operator::mouseReleaseEvent(QMouseEvent* event)
{
    if (event->button() == Qt::LeftButton) {
        for (Place*& place : _attack_places) {
            place->paintAttackArea();
        }
    }
}

为什么这两个函数内容完全一致呢？这里采用了一个相对简洁的写法，秘密在下头：

地块类这边接收信息，通过改变标志位，来改变 paintEvent 的效果。这正是按照状态显示不同信息时的通用手法。

void Place::paintAttackArea()
{
    _isPainted = !_isPainted; // 打印状态取反，相比于 if-else 更简洁
    update();
}

void Place::paintEvent(QPaintEvent*)
{
    if (_isPainted) {
        QPainter painter(this);
        painter.setPen(Qt::NoPen); // 取消描边
        painter.setBrush(QBrush(Qt::blue, Qt::FDiagPattern)); // 设置刷子格式为蓝色条纹状
        painter.drawRect(0, 0, 100, 100); // 画！
    }
}

void printLog(const QString& color, const QString& type, const QString& info, Map* map_init)
{
    //* 仅在第一次调用时传参 map_init，形成闭包
    static Map* map = map_init;
    if (map) {

时间管理者

众所周知，时间管理向来不是一件容易的事情，但由于之前的合理规划，我们的整个游戏逻辑是由一个时钟控制的：

  1. 只需要将 _timer 的触发时间间隔减半，就能实现二倍速。

  2. 只需要将 _timer 暂停，就能实现游戏的暂停。

为什么我只说“整个游戏逻辑”是由一个时钟控制的，而不是“整个游戏”呢？

区别在于，我力图达到图形层与逻辑层的尽量分离，每个单位重载了它的 timerEvent 以更新自身图像显示，这是全局 _timer 所不管的：

void Infected::timerEvent(QTimerEvent*)
{
    update();
}

startTimer(20); // 启动计时器。这句话写在 Infected 的构造函数中

也恰因此，当我们控制时间的时候，二倍速和暂停都不会影响到对象的动画正常播放。我认为这一点不算坏。

经过试验，图像更新会消耗大量资源。如果不异步处理，则会造成严重的卡顿。

并且，图像更新基本上是“无害”的，也即不会造成逻辑层的影响，便不会导致数据竞争，可以放心使用多线程处理。

游戏如何结束

最后的最后，请让我回收一个开头的伏笔：

在全局 update 函数中，我采用了一个 try-catch 语句来接收 GameOverException 类的异常。

把游戏结束定义为异常，这很怪，不是么？

但此举实际上好处多多，最重要的一点在于：

异常处理机制，使得游戏结束的信号可以迅速地脱离层层栈展开，以回到 update 函数内被马上处理掉。
这正是“游戏结束”与“异常”这两个看似毫不相关的概念之共性。

简单地写一下代码吧家人们~

class GameOverException : public QException {
public:
    virtual QString what() noexcept = 0; // what 函数必须是不抛出异常的
};

class WinGameException : public GameOverException {
public:
    QString what() noexcept override { return "You Win"; }
};

class LoseGameException : public GameOverException {
public:
    QString what() noexcept override { return "You Lose"; }
};

用基类引用类型捕获到异常之后，可以直接把 exception.what() 丢进 printLog 打印，而无需去管到底是玩家赢了还是输了。
