Static typing vs. Dynamic typing

  • 静态类型（static typing）的语言中，变量的类型在编译时便确定下来，那么使用变量的表达式的类型也就可以在编译时确定下来

  • 动态类型（dynamic typing）的语言，与之相反，变量和表达式的类型直到运行时才被确定下来

类型确定的时间并不影响程序的行为，其行为仍由变量事实上被赋予的值而决定——有很多静态类型的语言，在编译后就将类型丢出去了（C / Java, and C++ is not）

Static checking vs. Dynamic checking vs. No checking

静态类型是静态检查（static checking）的一种，也即在编译时检查 bug

因此，静态类型是 “消除 bug” 课题的第一个 idea

  • 静态检查可以认为是针对部分 错误 的
  • 动态检查则是针对 异常 的
  • 无检查则将尽力维持程序的运行（至于是否正常运行……）

无检查通过定义 正无穷 / NAN 等值，或返回 NULL 等来维持程序的运行，而在运行时并不向外抛出任何异常

Gradual typing

很多动态类型的语言现在也支持使用静态类型，这实际上使得渐进类型（gradual typing）得以存在，即同一程序的一部分使用静态类型，而另一部分使用动态类型

利用渐进类型，一个快速成型的动态类型小 demo ，逐渐成长为大型静态类型 project 的这一过程，就可以具有一条更平滑的曲线了

Software engineering

Hacking vs. Engineering

案：有风险的代码 vs. 软件工程

We've written some hacky code in this reading. Hacking is often marked by unbridled optimism:

  • Bad: writing lots of code before testing any of it.
  • Bad: keeping all the details in your head, assuming you'll remember them forever, instead of writing them down in your code.
  • Bad: assuming that bugs will be nonexistent or else easy to find and fix.

But software engineering is not hacking. Engineers are pessimists:

  • Good: write a little bit at a time, testing as you go. In a future class, we'll talk about test-first programming.
  • Good: document the assumptions that your code depends on.
  • Good: defend your code against stupidity – especially your own! Static checking helps with that.

再论可变性——值与引用

同上节课 const 处所述，改变内存地址的值，与改变变量映射到哪块内存区域中，是两个迥然不同的话题：

  • 可变对象与不可变对象之分，只在前者（value）
  • 而 const 考虑的问题，则在后者（reference）

可变性与不可变性的话题，自此便可细分，有了讨论的切入点

变量 是编译时实体，代表一块内存空间；对象 为运行时实体，代表一个数据单元

二者构成如下内存视图：变量名（变量的内存地址）-> 对象的值

按值传参

假设实参为 a，形参为 x，那么按值传参的过程可叙述为：

  • 创建变量 x 并分配新的内存空间
  • 将 a 所对应的对象拷贝到 x 分配到的内存空间中

这一过程实际上创建了一个起点为 x 的信息链路，作为原起点为 a 的信息链路之副本：

变量名 a（变量 a 的内存地址）-> 对象 M 的值

变量名 x（变量 x 的内存地址）-> 对象 M 的值

按引用传参

相对应地，按引用传参具有如下过程：

  • 创建变量 x 并将其映射到 a 的内存空间上

这一过程创建的信息链路如下所示，注意与按值传参的区别：

变量名 a（变量 a 的内存地址）-> 对象 M 的值

变量名 x（变量 a 的内存地址）-> 对象 M 的值

作为少有的允许按引用传参的语言，C++ 规定，一个引用类型的变量不能重新绑定到其他变量上，也即引用类型变量是隐式 const 的。这实际上维护了一定的不可变性

按值传参不一定是深拷贝

这是因为对象的值有可能是一个指针，如下所示：

变量名（变量的内存地址）

-> 对象 1 的值（对象 2 的内存地址）

-> 对象 2 的值

理论上，只要指针参数的阶数足够高，这个链条可以永续传递下去。而只有一直拷贝到此链条的尾部，才能在定义上算作是 深拷贝。否则即为 浅拷贝

Java 和 Python 都是按值传参的，但如果参数是引用类型的变量，都只能做到浅拷贝，这是因为其架构中变量所对应对象的值是一个隐式的指针之缘故

相反，按引用传参一定是浅拷贝，此理显见

引理：对不可变对象进行按值传参时……

任何对一个不可变对象进行的 操作 若是按值传参的，都不会改变实参变量本身

Containers

  • Array 对象接受隐式的列表构造方法
  • Set 对象可以通过构造函数显式调用一维数组实参来构造
  • Map 对象则可通过构造函数显式调用一维数组实参来构造，其元素类型必须为 [Key, Value]

Primitive vs. Object

TypeScript 中 基元 / 引用类型 的概念分野与 Java 基本相同

引用类型的对象的特征在于，它的值是一个隐式的、指向真正对象的一个指针

详细来说，取基元的对象可以直接得到它的值，而取引用类型的对象会先经过一个隐式的指针访问，再返回对应的真正对象的值：

基元：变量名 -> 对象 M 的值

引用类型对象：变量名 -> 指针 p（对象 N 的地址）-> 对象 N 的值

Insert element

容器添加元素时均为按值传参，因而具有上节所论按值传参所有性质：对基元是深拷贝，对引用类型对象是浅拷贝

TypeScript 的内嵌 Map / Set 为 HashMap / HashSet，这意味着它们的 key 对象需要同时满足两个条件：

  • Hashable
  • Immutable

任何基元自然满足上述条件，但如果想在容器里存储引用类型的对象，则要好好思量一番了

Visit element

  • 如果在遍历容器时不需要元素下标，则尽量使用范围 for 循环。因为常规 for 循环给 bug 预留了更多的生存空间

  • 如果试图访问容器中一个未被初始化的位置，则会返回 undefined

分为三类：

  • 形式化证明

Test-first programming

基本概念定义：

  • 模块（module）：一个系统中可以被独立设计、实现、测试、解释的单位，以下概念皆依特定模块而言
  • 说明（specification）：对模块行为的描述，包括以自然语言或程序设计语言写成的注释
  • 实现（implementation）：根据说明提供模块行为的程序实体
  • 用户（clients）：根据说明使用模块的程序实体
  • 测试样例（test case）：一个二元组，包含某个特定的输入及根据说明应有的输出，也即期望输出。测试亦属于模块用户
  • 测试集（test suite）：一组测试样例的集合

实现和用户的合法性全取决于其是否符合说明，用户对实现没有直接依赖性

测试驱动开发（test-first programming）的开发顺序为 说明 -> 测试 -> 实现。一旦实现代码通过所有测试用例，即标志着开发的基本完成

Systematic testing

系统化测试（systematic testing）意味着以一种规范化的方式选取测试样例，以建构兼具以下三种性质的测试集：

  • 正确性：测试集本身是一个合法的用户，且任何合法的实现应能通过所有的测试样例
  • 深透性：测试集应覆盖更多程序员易于写出的 bug，即应尽量使得有 bug 的实现不能通过所有测试样例
  • 小规模：这将使测试集更易于即时书写与迅速执行，并随说明更新而更新

上节所述三种他山石皆于深透性或小规模上有所不足，故不可用

测试者应当以 使得程序不能正常运行 为目标，以最野蛮粗暴的方式对待程序（诸君，我好兴奋啊x）

A good tester intentionally pokes at all the places the program might be vulnerable, so that those vulnerabilities can be eliminated.

Black box vs. Glass box

  • 黑盒测试（black box testing）是指测试样例依赖且仅依赖于说明的测试
  • 玻璃盒测试（glass box testing）是指测试样例不但依赖于说明，还参考于实现的测试，这是因为实现过程可能会引入通过说明所无法推断的边界条件。但测试样例仍然处于用户的地位，故不应对说明外的内容作任何假设

玻璃盒测试较黑盒测试而言更为彻底，但要时刻把握自身的 用户性 而不逾越，黑盒测试则无此疑问

下文主论黑盒测试，但玻璃盒测试亦同此法

Coverage

为了衡量测试集的深透性，引入覆盖（coverage）的概念

  • 语句覆盖（statement coverage）定义为每条语句都被至少一个测试样例执行过，为工业级代码之普遍标准
  • 分支覆盖（branch coverage）定义为每个 if / while 语句取 true 或者 false 的条件都被至少一个测试样例执行过，较语句覆盖更为严格

此二种覆盖均可由自动化工具加以检测

Choosing test cases by partitioning

基本思想：

把整个合法的输入域，按照输入的 相似程度 划分为不相交的子域，从每个子域中任取一个测试样例，即得到整个测试集

  • 测试集中每个样例都是不相似的，因而它足够的小
  • 测试集包含了所有种类的样例，因而它足够的深透

划分法的运用需要结合具体情况，对下文所述法则适当增删，然皆不离此处所述基本思想

划分法的数学解释

这一过程可以从纯函数的数学本质上加以考量，函数的输入域是由每个参数的输入域之交集所构成的 n 维空间：
<function>: (<Arg1_domain>, …, <ArgN_domain>) -> <Result_domain>

那么，输入的相似程度则可以定义为输出与输入之间构成的某种 简单函数关系，大体类似于分类讨论之形式

把每个输入域 <Argk_domain> 划分为子域集合 $ A_k = {D_{k0}, D_{k1}, …, D_{kM}} $，则每个测试样例映射到该输入域上，都一定取到且仅取到一个子域 $ D_{ki} $

那么，一个合理的测试集中的所有测试样例，在任一输入域上的映射，都应处于互不相交的子域之中，且该输入域上的每个子域都被大于等于一的、但尽量少的测试样例取到

依照此法，我们得到的测试集的规模，与最大的输入域子域集合的基数相等，但同时对每种情况都有一定强度的测试，即可同时满足优秀测试集的小规模和深透两个重要条件

但要注意的是，在这里对所有子域集合取笛卡儿积是不合理的。它虽然足够深透，但却会导致测试集大小和参数个数呈指数相关，很快其规模就会变得过于庞大而不适用

在划分中包含边界

边界情况（boundary）定义为两个划分域交界处的输入，应给予单独的、充分的测试

边界情况通常难以被 general 的代码解决，或由于 off-by-one mistakes 的易发性，而容易成为程序中 bug 的出现之处

Common boundaries：

  • 0, Tmax, Tmin
  • emptiness for collection types
  • the first and last element of a sequence

Unit testing

自动化的单元测试工具有利于测试集的维护与反复执行，使得测试集得以与模块一同不断迭代，保障渐进性开发中的正确性

mocha 是 JavaScript 语言的一种单元测试工具，故所有待测试的 TypeScript 文件均需被提前编译为对应的 JavaScript 文件

Export and import

由于测试文件必需访问待测试文件中的模块，这就涉及到 JavaScript 文件间名字的导入与导出：

JavaScript 中，名字的导入导出有两种规范，CommonJs 格式和 ES 格式

  • CommonJs 格式中，导入使用 exports 全局变量，导出使用 require 函数
  • ES 格式中，导入使用 export 关键词，导出使用 import 关键词

TypeScript 文件会默认以 CommonJs 格式进行编译，故在测试文件中也应以 CommonJs 格式的 require 导入模块

Example

以 ./test/sum.test.js 为例，测试代码示例如下：

const assert = require("assert");           // just search the name "assert" and import it
const sum_file = require('../out/sum.js');  // treat sum.js as a module to be imported
const sum = sum_file.sum;                   // get the name "sum" in file sum.js

describe("test sum", function () {
    /**
     * divide strategy:
     *  - empty list
     *  - non-empty list
     * 
     * to see the result as well as if the src list is changed
     */

    it("test empty list", function () {
        const src = [];
        assert.strictEqual(sum(src), 0);
        assert.strictEqual(src.length, 0);
    });
    it("test non-empty list", function () {
        const src = [3, 8, 9, 6, 5, 4, 7];
        const src_save = [3, 8, 9, 6, 5, 4, 7];
        assert.strictEqual(sum(src), 42);
        assert.deepEqual(src, src_save);
    });
});

经同步迭代，得到 ./src/sum.ts 代码如下：

/**
 * evaluate the sum of a given number list
 * @returns - the sum of all the numbers in the list;
 *          - if list is empty, return zero;
 */
export function sum(list: Array<number>): number {
    let result = 0;
    for (const val of list) {
        result += val;
    }
    return result;
}

Unit testing vs. Integration testing

  • 单元测试（unit test）定义为对单个模块隔离进行测试，上文所述皆属此部
  • 融合测试（integration test）定义为同时对多个模块一起进行的测试

融合测试是必要的，它能按照各模块之间的依赖关系对其进行校验。也正因此，对模块集合的融合测试必须在被其依赖的各个模块经单元测试通过后进行，以符合立即出错的原则

Iterative test-first programming

Iterate on a target module from scratch:

  • 1.Update specification for the module.
  • 2.Update tests that exercise the spec. If any problem is found, goto step 1.
  • 3.Update implementation until it can pass all tests. If any problem is found, goto step 1.

After the basic version is constructed, iterate the module while using it:

  • 4.If any optimizing changes of implementation is done, goto step 3.
  • 5.If any bug of this module is found, add it to the test set, then goto step 1.

Code Review

Objectives to learn:

  • code review: reading and discussing code written by somebody else
  • general principles of good coding: things you can look for in every code review, regardless of programming language or program purpose

Introduction

代码评审（code review）是对他人源码详细的、系统化的研究

代码评审的目的

  • 提升代码质量：发现 bug，预测可能的 bug，检查代码的清晰性，检查项目编程风格的一致性
  • 提升程序设计者的水平：促进其互相学习新的语言特性、新的技术，以及项目编程风格的改变

Don't repeat yourself (DRY)

代码拷贝会为内里潜藏的各种问题带来一份相近的副本，这会增加 bug 的修复难度

Comments where needed

好的程序设计者会以一种明智而审慎（judiciously）的方式书写注释

注释的功用：

  • 给予程序实体的行为以充分的限定性，这种注释可视作假设的书面化
  • 标明所引代码段的来源，网站名及作者名等

然而，注释不是对代码的重复，而是在代码的基础上对其进行的补充，无需对代码段已 明确 表明之信息再添注释

Fail fast

快速出错（fail fast）意味着程序应当尽早地让自身的 bug 显露出来，以便在其未产生连锁反应、且易于定位时即消灭之

快速出错的方式包括但不限于：

  • 充分利用类型检查机制（静态检查）
  • 检查输入的合法性并抛出异常（动态检查）
  • 单元测试

Avoid magic numbers

每个非平凡的字面量都被称作魔法数（magic number）。每个魔法数都应该被设计为一个有着明确名字的常值，大量的魔法数则可以数据结构的形式相组织

如果魔法数无法约简、且难以被赋予明确名字，则应附以适当注释以解释之，例如闰年算法中的 4 / 100 / 400 均属此类不可约简的魔法数

One purpose for each variable

Don’t reuse parameters, and don’t reuse variables.

Variables are not a scarce resource in programming. Introduce them freely, give them good names, and just stop using them when you stop needing them.

Use good names

好的名字应该是长且具有自文档性（self-descriptive）的

名字和注释同属代码中 说明 的类别，均能给予程序实体以限定性。因而如果名字已经具有充分的自文档性，则代码的可读性已能借此而保障，进而使得注释事实上是可以被取消的

Choose a good name

名字的书写形式要遵循所用语言的惯例。但通用的包括以大驼峰法书写的类名、以小驼峰法书写的函数名，变量名则按各语言所规：

  • Python：变量名-小写下划线
  • Java / JavaScript：全局不可变量-大写下划线，局部变量-小驼峰
  • C / C++：全局不可变量-大写下划线，局部变量-小写下划线

名字用词类型通常与程序实体的类型相关：

  • 函数名：动词短语
  • 类名和变量名：名词短语

命名时，应当选择简短而具有明确含义的词语

好的变量名应当能隐式或显式地标明其类型，好的函数名也应当能标明其返回值类型，这对动态类型语言更为关键

Use whitespace to help the reader

在表达式的元素间使用空格，在同组语句前使用一致的缩进

但不要在文本文件中留下 tab 字符，而要以空格代替。因其在不同编辑器中长度解释不同

Don't use global variables

全局可变量（global variables），即不具 const 属性的全局变量，是应当被避免的

避免全局可变量的方法包括但不限于：

  • 转化为函数参数
  • 转化为函数返回值
  • 封装成对象的数据成员

Functions should return results, not print them

在函数内同时完成计算和打印两项工作并不合适，这会导致计算结果的不可留存

整体而言，计算性函数属于底层架构，它只应强调其作为数学函数的性质

另一方面，只有顶层架构的代码才负责与用户交互

Avoid special-case code

能应付更多情况的 general code 可以减少 bug。因为它不仅减少了代码的复杂度，而且给予了输入更少的假设，从而加强了输入域的连续性

如非必要，决不在代码中添加排他条件

Sleep!

程序设计者的精神状态可能比具体的技术本身更会影响代码的优劣哦！

所以，睡~个~好~觉~吧~

Specifications

Objectives:

  • Understand preconditions and postconditions in function specifications, and be able to write correct specifications
  • Be able to write tests against a specification
  • Understand how to use exceptions

Introduction

说明（specification）是 API 的书面化。

在功能角度讲、对所述的程序实体具有限定性，在使用角度讲、对外界用户侧和内部实现侧均具有约定性。

从说明的使用角度演绎出来，则可知说明在时序上亦必须优先于用户侧和实现侧被完成。

说明的作用：

  • 说明对于概念的明晰，是维持一个良好体系的必要条件。
  • 说明对于概念的确定与限定，有助于缩小问题规模，利于简化用户和实现者思考的范围。
  • 说明作为具有中介性的抽象层，自然地具有解耦合的作用，避免用户和实现直接相互依赖。

结构良好的 + 概念明确的 -> 易于维护的 + 利于改变的

Behavioral equivalence

行为等价性（behavioral equivalence）指在 一定前提条件下，两个实现的所表现出的行为是否等价。

不同实现之间绝对的等价是不存在的。不过，实现是面向 API、进而面向用户的，用户只能观测到被 API 限定的实现，而无法（或不应该）直接观测到裸露的实现本身。因而，实现的行为等价性必须是在用户的可观测范围内，也即 API 限定范围内定义的，这样才有意义。

因此，行为等价性是 API 视域下对实现所提出的一种规范。程序设计体系中，三者实为一整体，不可孤立而看。

强大的前提条件或虚弱的后置条件，都能提高实现行为等价类的范围。

API 与函数的抽象属性——过程

从行为等价性一节可以看出，API 是从一个抽象的角度看待实现的。

一个实体意义上的函数（function），被抽象成了一个概念意义上的过程（procedure）。

抽象地说，描述 API 的说明乃由三部分组成：

  • 函数签名，包括函数名，参数的数量、类型和名称，以及返回值类型
  • 一个 require 从句，描述对参数的进一步限定
  • 一个 effects 从句，描述返回值、可能抛出的异常和函数副作用

也正因为 API 的抽象视域，它不能谈及任何关于函数具体实现的内容——API 本身在程序设计体系中，也是一个应当被严格界定的概念。

过程中的前提条件与后置条件

归纳而言，说明包含对这一 procedure 的两部分制约：
前提条件（precondition）、后置条件（postcondition），二者分别是以输入和输出为变元的谓词。

前提条件乃是在过程调用前为真的命题，满足它们是用户的责任。

后置条件（或称为从前提条件到后置条件的映射关系）乃是在过程调用后为真的命题，满足它们是实现的责任。

注意一点，后置条件是在前提条件被满足的情况下才要求成立的。如果前提条件不满足，则函数的行为不受后置条件限制。然而，作为一个习惯良好的程序设计者，此时程序应依照上一章所述的 Fail Fast 原则迅速告知用户异常。

前提条件范畴：

  • 函数签名中的参数类型及名称（以及实例方法中的隐式参数：this 指针）
  • 说明对每个参数范围的进一步规约，结果是参数只能取其类型范围的一个子集
  • 参数之间的相互制约

后置条件范畴：

  • 函数签名中的返回值类型
  • 返回值如何关联于参数
  • 哪些异常可能会被抛出，在何种情况下会被抛出
  • 参数是否可变，如果是，会被以何种方式改变

函数是过程的实例

自此，我们用论述的方法，把对程序的认识从函数这样的 程序实体 层面，提高到了过程这样的 抽象概念 层面。

但仍要分清：函数不是过程，函数是过程的实例。

说明的编程语言实现——注释与函数签名

在实现中，函数注释的简洁性尤为重要。

/**
 * <过程概述，包括前提条件和后置条件的部分属性>
 * Modifies <参数名> by <造成的改变>
 * 
 * @param <参数名> <与参数有关的其他前提条件>
 * @returns <与返回值有关的其他后置条件>
 * @throws <可能抛出的异常>
 */

函数注释中不应当包含任何与类型有关的信息，它们都应该被置于函数签名中。

Avoid null

作为良好的编程习惯之约定，函数的参数和返回值隐式地不应当在其任何位置含有 Null/None 或其他不具有本类型性质的值，除非经注释特别说明。

相反，空的容器对象具有本类型的所有性质，所以是隐式被允许的。

可变参数

除非经注释特别说明，我们通常认为参数是不会被改变的。

在编写 API 的时候，要特别注意多个指针类型的参数能否指向同一对象这一 corner case，它是隐式被允许的。

异常

异常是可预见的错误，这意味着：函数可能抛出的异常是后置条件的一部分。因此只有当前提条件满足时，这些异常才需要被说明。

通过减弱前提条件的限制性，我们可以扩宽函数的定义域，并对扩宽的那一部分子域引入异常：

if 满足 <旧前提>:
    旧后置条件
else if <以方式a不满足旧前提>:
    抛出异常a
else if ...

由于 TypeScript 是 JavaScript 的超集，其类型在编译后便会被抹除，因此在运行时可抛出与捕获的异常类型只有 Error。也即，不可能通过被捕获异常的类型来断定出现的错误，也不可能选择性地只捕获某些异常。

另外，异常可能会导致迭代的中断，此时可能只有部分元素被处理过了。把异常处理机制放到迭代的过程中，可改善这一点。

特殊的返回值

异常同时表征着用户难于处理的错误，这些错误严重到可以使得程序终止。

但对于另外一些不那么严重的错误，使用异常未免显得杀鸡牛刀。

尤其是对于“处理用户输入”这种应当给予“第二次机会”的过程来说，在用户输入错误时终止程序显然是有问题的做法。

对于那些不那么严重的错误，函数应当返回一个特殊值。在 TypeScript 中，使用 Union 类型的返回值，可以令函数在严格类型检查的情况下具备返回 undefined 或其他特殊值的能力。

Design Specifications

Objectives:

  • Understand underdetermined specs, and be able to identify and assess specs that are not deterministic.
  • Understand declarative vs. operational specs, and be able to write declarative specs.
  • Understand strength of preconditions, postconditions, and specs, and be able to compare spec strength.
  • Be able to write coherent, useful specifications of appropriate strength.

评价说明的量化指标

为设计说明，首先要确定好说明的标准。为确定标准，必有量化之指标。

为提出指标，必审视对象在体系中之位置，进而研究其作用。

现根据 API 在体系中的作用，给出评判说明优劣性的三个维度：

  • 输入对输出的决定性（deterministic）：对于一个给定的合法输入，函数的可能输出数量越少，函数说明的决定性越强。
  • 说明对实现的声明性（declarative）：说明是否只提及了输出与输入之间的抽象数学关系是什么，而并未谈及具体实现的操作性步骤。
  • 说明对实现的限定性（strong）：满足该说明的函数实现之集合越小，函数说明的限定性越强。

说明对实现的限定性

说明是实现的谓词。因为实现是黑匣子，说明对实现不直接产生规约，它经由前提条件规约输入，后置条件规约输出，从而达成对实现的描述与限制。

前提条件和后置条件也分别是输入和输出的谓词。

说明——作为谓词的限定性

说明对实现的限定性，实际上是指 说明 这个谓词对其谓词变元 实现 的限定性。

谓词的限定性定义为：

  • 当且仅当使得一个谓词为真的变元的取值区间，都严格地是另一个的子集时，我们称这个谓词具有更强的限定性。

说明 S2 具有不弱于 S1 的限定性，当且仅当以下条件同时满足：

  • S2 允许的输入范围是 S1 的超集（更弱的前提条件）
  • 在 S1 允许的输入范围内，S2 允许的输出范围是 S1 的子集（更强的后置条件）

若两条件之中仅满足一个，则 S2 与 S1 两个谓词是不可比的，同时体现在它们的变元取值集合之间不具有包含关系。

偏序关系视角的进一步解释

如果把说明的强弱定义为一种偏序关系，其取决于另两个偏序关系——前提条件和后置条件的强弱，则有：

S1 ≼ S2 <==> S2.pre ≼ S1.pre ∧ S1.post ≼ S2.post

限定性的实际意指

API 在整个体系中起到一个规约的作用，它是最先被确定的实体，进而成为了用户侧和实现侧的依靠：

  • 它为实现带来了自由性，实现只需满足 API 的要求即可。在这些要求允许的范围内，程序设计者可以自由选择其实现方式。
  • 它为用户带来了确定性，用户可以充分信任 API，而不必担心自己调用的过程不满足其需求。

API 限定性的增强，意味着实现侧自由性的削减，也意味着用户侧确定性的增强。满足要求的实现是原先的子集，可以使用该 API 的用户却变成了原先的超集。

一个简单的推论是：

如果 S2 比 S1 具有更强的限定性，则可以在不影响用户代码的情况下，将说明 S1 改为 S2。

设计更好的说明

由于函数的行为全由说明所断言，因此设计更好的说明实际上蕴含了设计更好的函数。

还依本章开头时所提的三点，给出如下建议：

  • 更强的决定性：每个结果所表示的情况数应尽可能地少，进而每个结果包含了更多有效信息。决定性体现了函数的实际 能力。
  • 绝对逻辑的声明性：说明所描述的过程应当自成一个独立单元，它的行为应当 make sense。也就是说，一个说明不应当由多个相互之间没有逻辑关联的断言组成。
  • 适当的限定性：见上文。

前提条件应足够紧实

前提条件界定了过程的合法输入区间，使得实现更为简单。

然而，这种简化不是偷懒的工具。不满足前提条件的情况，应当是实现者在技术上难于检测、或需要花费大量资源才能检测的。

如果前提条件越出了这个范围，也就是给予输入更多的限制，那么当用户的输入不满足这些限制时，实现侧就难以察觉并导致程序的错误。这样的错误通常是难于准确定位的。因此，过高的前提条件限制是不佳的。

如上章所述，可以通过把冗余的前提转换成后置条件中特殊值或异常的方式，来削弱前提条件，使之足够地紧实。这是遵循 fail fast 的原则。

以上所论是通常的情况，但由于检测特殊情况与抛出异常对实现者也是一种负担，对于那些只在内部调用、不向外开放接口的函数，可以适当地具有强前提条件。

API 作为抽象概念的界定

在每个程序世界中，最先诞生的不是 Hello World 的程序实体，而是 Hello World 过程的抽象概念。而这一抽象概念又必须经 API ——以程序为变元的谓词——所断言，才能被人类认知。

我认为，这才是抽象的原初意义：

抽象一种思维方式，使得我们可以用一个概念来描述一个程序实体，用一组已知的概念来界定一个未知的概念。

在抽象过程中产生的副作用是，对于同样的事物，我们可以用更少的信息来表示。这常常带给人们一种错觉，就好像抽象就是为了控制复杂度而生的一样，这种错觉实际上掩盖了其更为深刻的内涵。

API 在程序设计体系中的重要位置

API 于程序诞生之初诞生，成为了整个程序的第一个着力点。

其他程序实体，都为 API 所断言，进而依靠于它。可以说，API 是整个程序的核心。

既然 API 直接影响着程序的好坏，若要改变，也当由 API 始。

Mutability

Objectives:

  • Understand mutability and mutable objects
  • Identify aliasing and understand the dangers of mutability
  • Use immutability to improve correctness, clarity, changeability

变量名层级的不可变性

TypeScript 为每个内置容器类都定义了相对应的只读接口，程序员可以利用这些接口声明变量。

虽然这些变量所绑定的对象之动态类型仍是可变的，但依赖静态类型检查机制，用户无法在通过以只读接口声明的变量来改变它们。

但这种依靠静态类型检查机制获得的不可变性是有缺陷的：

它们是对变量名的检查，而不是对对象本身的检查。

如果通过赋值等操作，这些对象被绑定到了其他以读写接口声明的变量名下，那么它们将仍可以通过这些变量访问并被改变。

可变对象的风险

可变对象的隐患只有在其 别名（aliases）在不同地方出现 时才会显露出来。

在对象与多个名字相绑定时，不同名字的使用者可能在互不知情的情况下改变该对象的值。

最常见的是在函数调用的过程中：

  • 传参时：利用良好的说明风格，程序设计者固然可以把参数类型设为可变对象。但此一规则不具有强制性，可变参数本身就为 bug 的出现带来了可能。
  • 返回时：不要在返回值中传递函数缓存区中可变对象的引用，可以通过返回时的防御性拷贝来防止这一点。

而以上问题，都可以通过在最初就把对象设计成不可变的而得以防止，毕竟：

消除 bug 的最有效方式是令 bug 在设计时就没有存在的空间。

可变对象与契约

对象在程序结构中的广泛传播，同时要求了其契约的广泛传播。

对于不可变对象，并无此顾虑，因为一处错误的使用不会影响到其他用户。

而于可变对象而言，在一个地方改变对象、然后通知其他所有用户是事实上难于实现的。在此种前提下，该可变对象的不同用户之间必须达成对可变性的某种契约。这种契约被迫要在对象的整个生命周期内、在多处使用该对象的用户手中保存。

一个不具有 局部作用域 的、复杂的契约，会大大增加程序设计者的脑力消耗和出错概率。

在实际使用上，我们给出如下提议：

  • 如果一个对象不需频繁修改，那么它最好被设计成不可变的，这还有利于它在不同程序实体、不同时间段内上被广泛共享
  • 如果一个对象要被频繁修改、且在时空上都能保持良好的局部作用域，那么它可以被设计成可变的

Avoid Debugging

Objective:

We are going to look at ways to write code that either avoids debugging entirely,
or at least makes it easy when we have to do it.

First defense: make bugs impossible

御敌于千里之外，包括前文已述的对静态检查和不可变性的利用。

Second defense: localize bugs

遵循 Fast Fail 的原则：

  • 尽量检查函数的前提条件是否被满足——这是被称作防御性编程（defensive programming）的设计方法。
  • 另一方面，可在返回前检查后置条件是否被满足，这点被称作自我查验（self check）。

所有前提条件和后置条件都可以被称作 不变量（invariant）。它不一定是某一个值，而是当前时空下满足的某种性质或联系。

把工作量局限在时空的局部范围内。

Assertion

assert 是对 检查条件 这一过程的封装函数，用于开发和调试。它也像 const 等关键词或注释一样，属于假设的书面化。

由于 assert 的使用有可能会使代码变得杂乱，因此程序员应当明智地使用它们：

  • 非平凡的：assert 是注释的一种，所以与注释一样，不要做平凡的校验——程序设计的基础是对数学理论和编程语言固有设施的完全信任。
  • 针对内部的固有问题：assert 的作用是指出程序中的 bug，这暗示着该 bug 是由程序的内部结构导致的；对于来自程序外部的输入，不要 assert——使用特定返回值或抛出异常来解决这一点。
  • 无副作用的：assert 是在调试时使用的，这意味着它们会在 release 版本中会被默认关闭。因此有无 assert 不应当影响程序的正确性，即 assert 不应当引发副作用。
  • 无人区校验：在程序不应当到达的地方，如 switch 语句的 default 段等，可以作 assert 进行校验。

时间上：递进式开发

递进式开发（incremental development）是从开发的动态过程层面使 bug 局部化的方法，包括前文已提及的单元测试和回返性测试的方法。

空间上：增强模块化和封装性

即使是设计最优良的 API，也仍然属于契约的范畴。只要是契约，就使得程序的正确性部分地依赖于用户方的自觉而成立。

我们无法保证作为人类的用户不犯任何错误，因此任何契约也都应当把自身的可见性限制在狭窄的局部。

避免契约的过度传播，我们使用模块化和封装来做到这一点。

把每个过程的可见性限定在很紧致的范围内：

  • 把尽可能少的接口，即 public 成员，暴露给用户。

把每个变量的作用域限定在很紧致的范围内：

  • 不要使用全局可变量。
  • 只有当首次需要一个变量时，才声明它。
