Static typing vs. Dynamic typing

  • 静态类型（static typing）的语言中，变量的类型在编译时便确定下来，那么使用变量的表达式的类型也就可以在编译时确定下来

  • 动态类型（dynamic typing）的语言，与之相反，变量和表达式的类型直到运行时才被确定下来

类型确定的时间并不影响程序的行为，其行为仍由变量事实上被赋予的值而决定——有很多静态类型的语言，在编译后就将类型丢出去了（C / Java, and C++ is not）

Static checking vs. Dynamic checking vs. No checking

静态类型是静态检查（static checking）的一种，也即在编译时检查 bug

因此，静态类型是 “消除 bug” 课题的第一个 idea

  • 静态检查可以认为是针对部分 错误 的
  • 动态检查则是针对 异常 的
  • 无检查则将尽力维持程序的运行（至于是否正常运行……）

无检查通过定义 正无穷 / NAN 等值，或返回 NULL 等来维持程序的运行，而在运行时并不向外抛出任何异常

Gradual typing

很多动态类型的语言现在也支持使用静态类型，这实际上使得渐进类型（gradual typing）得以存在，即同一程序的一部分使用静态类型，而另一部分使用动态类型

利用渐进类型，一个快速成型的动态类型小 demo ，逐渐成长为大型静态类型 project 的这一过程，就可以具有一条更平滑的曲线了

Tips about grammer

Details

  • TypeScript 中可以指定变量类型为 any，则编译器会跳过所有与之相关的静态类型检查。所有包含类型为 any 的变量的表达式的类型也为 any

  • 在 TypeScript 中，用 for(let i of <iterable>) 来遍历容器中的元素，案 Map 的元素即为形式为 [key, value] 的二元数组。在任何情况下不要使用 for(let i in <iterable>)

  • C++ 和 Python 里面都没有从其他类型到 string 的隐式类型转换，而 Java 和 JavaScript 是有的

  • null 类型和 undefined 同为 TypeScript 的基本类型，但其各自自成一派，无法赋值给其他任何非属 null 或 undefined 的类型

Mutable variable and const identifier

可变性是不可避免的罪恶！好的程序员都应该试着去尽量避免可变性

对尽可能多的变量施以 const 标识符，这将是对编译器和其他程序员很好的注释

但要注意，const 标识符只能保证变量名对变量地址的引用保持不变。而如果变量地址上存储的是一个可变对象，则 const 无所助益，就像 Java 中的 final 标识符那样

Comment for functions

函数的注释非常重要，其主要功能在于对函数功能的进一步阐明，给予函数作为一种抽象概念“操作”的充分限定性

何谓“进一步”？

是在函数原型的基础上，对函数原型所没有包含的、但却于调用者关系密切的信息的阐明

注释三要点：内容简洁、语义明确、描述精当

Why do we need to write down our assumptions?

Because programming is full of them, and if we don't write them down, we won't remember them, and other people who need to read or change our programs later won't know them. They'll have to guess.

Software engineering

Hacking vs. Engineering

案：有风险的代码 vs. 软件工程

We've written some hacky code in this reading. Hacking is often marked by unbridled optimism:

  • Bad: writing lots of code before testing any of it.
  • Bad: keeping all the details in your head, assuming you'll remember them forever, instead of writing them down in your code.
  • Bad: assuming that bugs will be nonexistent or else easy to find and fix.

But software engineering is not hacking. Engineers are pessimists:

  • Good: write a little bit at a time, testing as you go. In a future class, we'll talk about test-first programming.
  • Good: document the assumptions that your code depends on.
  • Good: defend your code against stupidity – especially your own! Static checking helps with that.

Goal of 6.031

软件工程是着眼于未来的，因为软件是一个不断需要被改变的东西，因而它必须以准备被改变的形式去建构

也正因软件工程的未来性，软件的每一子结构也必须有良好的 自我限定性。

——事实上，从最初几行 demo 开始，每一点增量更新都可以看作是一个 change

本课三目标：

  • Safe from bugs.
  • Easy to understand.
  • Ready for change.

It's worth considering every language feature, every programming practice, every design pattern that we study in this course, and understanding how they relate to the Big Three.

Review call-by-value

假设实参为 a，形参为 x，那么传参过程可叙述为：

  • 创建变量 x 并分配新的内存空间
  • 将 a 所对应的对象拷贝到 x 分配到的内存空间中

这一过程实际上创建了一个起点为 x 的信息链路，作为原起点为 a 的信息链路之副本：

变量名 a（变量 a 的内存地址）-> 对象 M 的值

变量名 x（变量 x 的内存地址）-> 对象 M 的值

Review call-by-reference

相对应地，按引用传参具有如下过程：

  • 创建变量 x 并将其映射到 a 的内存空间上

这一过程创建的信息链路如下所示，注意比按值传参短了一步：

变量名 a（变量 a 的内存地址）

变量名 x（变量 a 的内存地址）

作为少有的允许按引用传参的语言，C++ 规定，一个引用类型的变量不能重新绑定到其他变量上，也即引用类型变量是隐式 const 的。这实际上维护了一定的不可变性

Call-by-value != Deep copy

案对象的值有可能是一个指针，如下所示：

变量名（变量的内存地址）-> 对象 1 的值（对象 2 的内存地址）-> 对象 2 的值

理论上，只要指针参数的阶数足够高，这个链条可以永续传递下去。而只有一直拷贝到此链条的尾部，才能在定义上算作是 深拷贝。否则即为 浅拷贝
Java 和 Python 都是按值传参的，但如果参数是引用类型的变量，都只能做到浅拷贝，这是因为其架构中变量所对应对象的值是一个隐式的指针之缘故

相反，按引用传参一定是浅拷贝，此理显见

Lemma: when pass a immutable object by value …

综上所述，任何对一个不可变对象进行的 操作 若是按值传参的，都不会改变实参变量本身

Containers

Arrays, Maps, and Sets included.

  • Array 对象接受隐式的列表构造方法
  • Set 对象可以通过构造函数显式调用一维数组实参来构造
  • Map 对象则可通过构造函数显式调用二维数组实参来构造，其中内层数组的形式必须为 [key, value]

Primitive vs. Object

案：TypeScript 中 基元 / 引用类型 的概念分野与 Java 基本相同。如 number 与 string 即属基元，而 Number, String, Array 等则属包装类型的对象

引用类型的对象的特征在于，它的值是一个隐式的、指向真正对象的一个指针

详细来说，取基元的对象可以直接得到它的值，而取引用类型的对象会先经过一个隐式的指针访问，再返回对应的真正对象的值：

基元：变量名（变量的内存地址）-> 对象 M 的值

引用类型对象：变量名（变量的内存地址）-> 指针 p（对象 N 的地址）-> 对象 N 的值

Insert element

容器添加元素时均为按值传参，因而具有上节所论按值传参所有性质。对基元是深拷贝，对引用类型对象是浅拷贝

TypeScript 的内嵌 Map / Set 为 HashMap / HashSet，这意味着它们的 key 对象需要同时满足两个条件：

  • hashable
  • immutable

任何基元自然满足上述条件，但如果想在容器里存储引用类型的对象，则要好好思量一番了

Visit element

  • 如果在遍历容器时不需要元素下标，则尽量使用范围 for 循环。因为常规 for 循环给 bug 预留了更多的生存空间

  • 如果试图访问容器中一个未被初始化的位置，则会返回 undefined

  • TypeScript 中数组可以在任意下标存储对象（甚至是负数下标），即使对象的存储并不连续。但这样的情况并不应多见，程序员在更多时候应当给予 Array 以 抽象数据结构 list 的限定

More grammer about variable

declare variable

  • TypeScript 支持变量类型的自动推断，如同 C++ 中 auto 关键字那样。但应极其谨慎地使用，否则不能给人以充分的 注释

  • 变量的性质部分地 受限 于声明时所用的关键字：

let 关键字声明的变量具有块作用域，且应可能被重新赋值

const 关键字声明的变量具有块作用域，且不能被重新赋值

var 关键字声明的变量具有函数作用域，这点同 Python 一致，给予了变量更弱的限定性，不应被使用

Record type and Destructuring assignment

记录类型与对象解构是一组对称的语法，用于处理一组临时的对象集合

  • 记录类型（record type），类似于“匿名结构体”，用于把一组变量聚合到一个记录类型的变量上，或作为函数返回多个变量的方式。可视为 打包
  • 对象解构（destructuring assignment）则可把记录类型的变量集合拆开，按顺序分别为一组新声明的同名变量赋初值。可视为 解包

API

程序依靠不同的 API（application programming interface），来在不同的上下文中运行。程序的编写者要尽量利用通用的 API 编写代码，以使得程序与运行环境无关

API 对 程序实体 的描述应当给予其充分的限定性，以至于用户可以仅凭说明来使用它们
