Static typing vs. Dynamic typing

  • 静态类型（static typing）的语言中，变量的类型在编译时便确定下来，那么使用变量的表达式的类型也就可以在编译时确定下来

  • 动态类型（dynamic typing）的语言，与之相反，变量和表达式的类型直到运行时才被确定下来

类型确定的时间并不影响程序的行为，其行为仍由变量事实上被赋予的值而决定——有很多静态类型的语言，在编译后就将类型丢出去了（C / Java, and C++ is not）

Static checking vs. Dynamic checking vs. No checking

静态类型是静态检查（static checking）的一种，也即在编译时检查 bug

因此，静态类型是 “消除 bug” 课题的第一个 idea

  • 静态检查可以认为是针对部分 错误 的
  • 动态检查则是针对 异常 的
  • 无检查则将尽力维持程序的运行（至于是否正常运行……）

无检查通过定义 正无穷 / NAN 等值，或返回 NULL 等来维持程序的运行，而在运行时并不向外抛出任何异常

Gradual typing

很多动态类型的语言现在也支持使用静态类型，这实际上使得渐进类型（Gradual typing）得以存在，即同一程序的一部分使用静态类型，而另一部分使用动态类型

利用渐进类型，一个快速成型的动态类型小 demo ，逐渐成长为大型静态类型 project 的这一过程，就可以具有一条更平滑的曲线了

Tips about grammer

Details

  • TypeScript 中可以指定变量类型为 any，则编译器会跳过所有与之相关的静态类型检查。所有包含类型为 any 的变量的表达式的类型也为 any

  • 在 TypeScript 中，用 for (let i in <iterable>) 来遍历 key，用 for (let i of <iterable>) 来遍历 value。案数组的 key 即其下标

  • C++ 和 Python 里面都没有从其他类型到 string 的隐式类型转换，而 Java 和 JavaScript 是有的

  • null 类型和 undefined 同为 TypeScript 的基本类型，但其各自自成一派，无法赋值给其他任何非属 null 或 undefined 的类型

Mutable variable and const identifier

可变性是不可避免的罪恶！好的程序员都应该试着去尽量避免可变性

对尽可能多的变量施以 const 标识符，这将是对编译器和其他程序员很好的注释

但要注意，const 标识符只能保证变量名对变量地址的引用保持不变。而如果变量地址上存储的是一个可变对象，则 const 无所助益，就像 Java 中的 final 标识符那样

Comment for functions

函数的注释非常重要，其主要功能在于对函数功能的进一步阐明，给予函数作为一种抽象概念“操作”的充分限定性

何谓“进一步”？

是在函数原型的基础上，对函数原型所没有包含的、但却于调用者关系密切的信息的阐明

注释三要点：内容简洁、语义明确、描述精当

Why do we need to write down our assumptions?

Because programming is full of them, and if we don't write them down, we won't remember them, and other people who need to read or change our programs later won't know them. They'll have to guess.

Software engineering

Hacking vs. engineering

案：有风险的代码 vs. 软件工程

We've written some hacky code in this reading. Hacking is often marked by unbridled optimism:

  • Bad: writing lots of code before testing any of it.
  • Bad: keeping all the details in your head, assuming you'll remember them forever, instead of writing them down in your code.
  • Bad: assuming that bugs will be nonexistent or else easy to find and fix.

But software engineering is not hacking. Engineers are pessimists:

  • Good: write a little bit at a time, testing as you go. In a future class, we'll talk about test-first programming.
  • Good: document the assumptions that your code depends on.
  • Good: defend your code against stupidity – especially your own! Static checking helps with that.

Goal of 6.031

软件工程是着眼于未来的，因为软件是一个不断需要被改变的东西，因而它必须以准备被改变的形式去建构

——事实上，从最初几行 demo 开始，每一点增量更新都可以看作是一个 change

本课三目标：

  • Safe from bugs.
  • Easy to understand.
  • Ready for change.

It's worth considering every language feature, every programming practice, every design pattern that we study in this course, and understanding how they relate to the Big Three.
