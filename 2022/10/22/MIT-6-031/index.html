Static typing vs. Dynamic typing

  • 静态类型（static typing）的语言中，变量的类型在编译时便确定下来，那么使用变量的表达式的类型也就可以在编译时确定下来

  • 动态类型（dynamic typing）的语言，与之相反，变量和表达式的类型直到运行时才被确定下来

类型确定的时间并不影响程序的行为，其行为仍由变量事实上被赋予的值而决定——有很多静态类型的语言，在编译后就将类型丢出去了（C / Java, and C++ is not）

Static checking vs. Dynamic checking vs. No checking

静态类型是静态检查（static checking）的一种，也即在编译时检查 bug

因此，静态类型是 “消除 bug” 课题的第一个 idea

  • 静态检查可以认为是针对部分 错误 的
  • 动态检查则是针对 异常 的
  • 无检查则将尽力维持程序的运行（至于是否正常运行……）

无检查通过定义 正无穷 / NAN 等值，或返回 NULL 等来维持程序的运行，而在运行时并不向外抛出任何异常

Gradual typing

很多动态类型的语言现在也支持使用静态类型，这实际上使得渐进类型（gradual typing）得以存在，即同一程序的一部分使用静态类型，而另一部分使用动态类型

利用渐进类型，一个快速成型的动态类型小 demo ，逐渐成长为大型静态类型 project 的这一过程，就可以具有一条更平滑的曲线了

Software engineering

Hacking vs. Engineering

案：有风险的代码 vs. 软件工程

We've written some hacky code in this reading. Hacking is often marked by unbridled optimism:

  • Bad: writing lots of code before testing any of it.
  • Bad: keeping all the details in your head, assuming you'll remember them forever, instead of writing them down in your code.
  • Bad: assuming that bugs will be nonexistent or else easy to find and fix.

But software engineering is not hacking. Engineers are pessimists:

  • Good: write a little bit at a time, testing as you go. In a future class, we'll talk about test-first programming.
  • Good: document the assumptions that your code depends on.
  • Good: defend your code against stupidity – especially your own! Static checking helps with that.

Call-by-value

假设实参为 a，形参为 x，那么按值传参的过程可叙述为：

  • 创建变量 x 并分配新的内存空间
  • 将 a 所对应的对象拷贝到 x 分配到的内存空间中

这一过程实际上创建了一个起点为 x 的信息链路，作为原起点为 a 的信息链路之副本：

变量名 a（变量 a 的内存地址）-> 对象 M 的值

变量名 x（变量 x 的内存地址）-> 对象 M 的值

Call-by-reference

相对应地，按引用传参具有如下过程：

  • 创建变量 x 并将其映射到 a 的内存空间上

这一过程创建的信息链路如下所示，注意与按值传参的区别：

变量名 a（变量 a 的内存地址）-> 对象 M 的值

变量名 x（变量 a 的内存地址）-> 对象 M 的值

作为少有的允许按引用传参的语言，C++ 规定，一个引用类型的变量不能重新绑定到其他变量上，也即引用类型变量是隐式 const 的。这实际上维护了一定的不可变性

Call-by-value != Deep copy

案：对象的值有可能是一个指针，如下所示：

变量名（变量的内存地址）

-> 对象 1 的值（对象 2 的内存地址）

-> 对象 2 的值

理论上，只要指针参数的阶数足够高，这个链条可以永续传递下去。而只有一直拷贝到此链条的尾部，才能在定义上算作是 深拷贝。否则即为 浅拷贝

Java 和 Python 都是按值传参的，但如果参数是引用类型的变量，都只能做到浅拷贝，这是因为其架构中变量所对应对象的值是一个隐式的指针之缘故

相反，按引用传参一定是浅拷贝，此理显见

Primitive vs. Object

案：TypeScript 中 基元 / 引用类型 的概念分野与 Java 基本相同

引用类型的对象的特征在于，它的值是一个隐式的、指向真正对象的一个指针

详细来说，取基元的对象可以直接得到它的值，而取引用类型的对象会先经过一个隐式的指针访问，再返回对应的真正对象的值：

基元：变量名 -> 对象 M 的值

引用类型对象：变量名 -> 指针 p（对象 N 的地址）-> 对象 N 的值

Insert element

容器添加元素时均为按值传参，因而具有上节所论按值传参所有性质。对基元是深拷贝，对引用类型对象是浅拷贝

TypeScript 的内嵌 Map / Set 为 HashMap / HashSet，这意味着它们的 key 对象需要同时满足两个条件：

  • Hashable
  • Immutable

任何基元自然满足上述条件，但如果想在容器里存储引用类型的对象，则要好好思量一番了

Visit element

  • 如果在遍历容器时不需要元素下标，则尽量使用范围 for 循环。因为常规 for 循环给 bug 预留了更多的生存空间

  • 如果试图访问容器中一个未被初始化的位置，则会返回 undefined

  • TypeScript 中数组可以在任意下标存储对象（甚至是负数下标），即使对象的存储并不连续。但这样的情况并不应多见，程序员在更多时候应当给予 Array 以抽象数据结构 list 的限定

Record type and Destructuring assignment

记录类型与对象解构是一组对称的语法，用于处理一组临时的对象集合

  • 记录类型（record type），类似于“匿名结构体”，用于把一组变量聚合到一个记录类型的变量上，或作为函数返回多个变量的方式。可视为 打包
  • 对象解构（destructuring assignment）则可把记录类型的变量集合拆开，按顺序分别为一组新声明的同名变量赋初值。可视为 解包

Validation

正确性确认（validation） 定义为通过揭露程序中的问题，来增强程序员对程序正确性的自信

分为三类：

  • Verification: Constructing a formal proof that a program is correct.
  • Code review: Having somebody else carefully read your code, and reason informally about it.
  • Testing: Running the program on carefully selected inputs and checking the results.

通过正确性确认来完全规避 bug 是几不可能的，然而，10 缺陷 / 千行代码 的出厂错误率已能达到工业软件之标准

Problems of existing ways

  • 完全测试：对输入的全集进行测试，过于昂贵
  • 随机测试：随机挑选样例进行测试，可靠性过低
  • 抽样测试：抽样测试的有效性依赖于缺陷在检测对象上的分布是连续或均匀的，而错误输出在输入域上的分布却是离散且突变的

因此，需要有一种适应于软件工程特性的系统化测试方法

Test-first programming

基本概念定义：

  • 模块（module）：一个系统中可以被独立设计、实现、测试、解释的单位，以下概念皆依特定模块而言
  • 说明（specification）：对模块行为的描述，包括以自然语言或程序设计语言写成的注释
  • 实现（implementation）：根据说明提供模块行为的程序实体
  • 用户（clients）：根据说明使用模块的程序实体
  • 测试样例（test case）：一个二元组，包含某个特定的输入及根据说明应有的输出，也即期望输出。测试亦属于模块用户
  • 测试集（test suite）：一组测试样例的集合

实现和用户的合法性全取决于其是否符合说明，用户对实现没有直接依赖性

测试驱动开发（test-first programming）的开发顺序为 说明 -> 测试 -> 实现。一旦实现代码通过所有测试用例，即标志着开发的基本完成

Systematic testing

系统化测试（systematic testing）意味着以一种规范化的方式选取测试样例，以建构兼具以下三种性质的测试集：

  • 正确性：测试集本身是一个合法的用户，且任何合法的实现应能通过所有的测试样例
  • 深透性：测试集应覆盖更多程序员易于写出的 bug，即应尽量使得有 bug 的实现不能通过所有测试样例
  • 小规模：这将使测试集更易于即时书写与迅速执行，并随说明更新而更新

上节所述三种他山石皆于深透性或小规模上有所不足，故不可用

测试者应当以 使得程序不能正常运行 为目标，以最野蛮粗暴的方式对待程序（诸君，我好兴奋啊x）

A good tester intentionally pokes at all the places the program might be vulnerable, so that those vulnerabilities can be eliminated.

Black box vs. Glass box

  • 黑盒测试（black box testing）是指测试样例依赖且仅依赖于说明的测试
  • 玻璃盒测试（glass box testing）是指测试样例不但依赖于说明，还参考于实现的测试，这是因为实现过程可能会引入通过说明所无法推断的边界条件。但测试样例仍然处于用户的地位，故不应对说明外的内容作任何假设

玻璃盒测试较黑盒测试而言更为彻底，但要时刻把握自身的 用户性 而不逾越，黑盒测试则无此疑问

下文主论黑盒测试，但玻璃盒测试亦同此法

Coverage

为了衡量测试集的深透性，引入覆盖（coverage）的概念

  • 语句覆盖（statement coverage）定义为每条语句都被至少一个测试样例执行过，为工业级代码之普遍标准
  • 分支覆盖（branch coverage）定义为每个 if / while 语句取 true 或者 false 的条件都被至少一个测试样例执行过，较语句覆盖更为严格

此二种覆盖均可由自动化工具加以检测

Choosing test cases by partitioning

基本思想：

把整个合法的输入域，按照输入的 相似程度 划分为不相交的子域，从每个子域中任取一个测试样例，即得到整个测试集

  • 测试集中每个样例都是不相似的，因而它足够的小
  • 测试集包含了所有种类的样例，因而它足够的深透

划分法的运用需要结合具体情况，对下文所述法则适当增删，然皆不离此处所述基本思想

Mathematical explaination of partitioning

这一过程可以从纯函数的数学本质上加以考量，函数的输入域是由每个参数的输入域之交集所构成的 n 维空间：
<function>: (<Arg1_domain>, …, <ArgN_domain>) -> <Result_domain>

那么，输入的相似程度则可以定义为输出与输入之间构成的某种 简单函数关系，大体类似于分类讨论之形式

把每个输入域 <Argk_domain> 划分为子域集合 $ A_k = {D_{k0}, D_{k1}, …, D_{kM}} $，则每个测试样例映射到该输入域上，都一定取到且仅取到一个子域 $ D_{ki} $

那么，一个合理的测试集中的所有测试样例，在任一输入域上的映射，都应处于互不相交的子域之中，且该输入域上的每个子域都被大于等于一的、但尽量少的测试样例取到

依照此法，我们得到的测试集的规模，与最大的输入域子域集合的基数相等，但同时对每种情况都有一定强度的测试，即可同时满足优秀测试集的小规模和深透两个重要条件

但要注意的是，在这里，对所有子域集合取笛卡儿积是不合理的。它虽然足够深透，但却会导致测试集大小和参数个数呈指数相关，很快其规模就会变得过于庞大而不适用

Include boundaries in the partition

边界情况（boundary）定义为两个划分域交界处的输入，应给予单独的、充分的测试

边界情况通常难以被 general 的代码解决，或由于 off-by-one mistakes 的易发性，而容易成为程序中 bug 的出现之处

Common boundaries：

  • 0, Tmax, Tmin
  • emptiness for collection types
  • the first and last element of a sequence

Unit testing

自动化的单元测试工具有利于测试集的维护与反复执行，使得测试集得以与模块一同不断迭代，保障渐进性开发中的正确性

mocha 是 JavaScript 语言的一种单元测试工具，故所有待测试的 TypeScript 文件均需被提前编译为对应的 JavaScript 文件

Export and import

由于测试文件必需访问待测试文件中的模块，这就涉及到 JavaScript 文件间名字的导入与导出：

JavaScript 中，名字的导入导出有两种规范，CommonJs 格式和 ES 格式

  • CommonJs 格式中，导入使用 exports 全局变量，导出使用 require 函数
  • ES 格式中，导入使用 export 关键词，导出使用 import 关键词

TypeScript 文件会默认以 CommonJs 格式进行编译，故在测试文件中也应以 CommonJs 格式的 require 导入模块

Example

以 ./test/sum.test.js 为例，测试代码示例如下：

const assert = require("assert");           // just search the name "assert" and import it
const sum_file = require('../out/sum.js');  // treat sum.js as a module to be imported
const sum = sum_file.sum;                   // get the name "sum" in file sum.js

describe("test sum", function () {
    /**
     * divide strategy:
     *  - empty list
     *  - non-empty list
     * 
     * to see the result as well as if the src list is changed
     */

    it("test empty list", function () {
        const src = [];
        assert.strictEqual(sum(src), 0);
        assert.strictEqual(src.length, 0);
    });
    it("test non-empty list", function () {
        const src = [3, 8, 9, 6, 5, 4, 7];
        const src_save = [3, 8, 9, 6, 5, 4, 7];
        assert.strictEqual(sum(src), 42);
        assert.deepEqual(src, src_save);
    });
});

经同步迭代，得到 ./src/sum.ts 代码如下：

/**
 * evaluate the sum of a given number list
 * without change the content of it
 * @param list - a number list
 * @returns - the sum of all the numbers in the list;
 *          - if list is empty, return zero;
 *          - the precise of output is depend on type 'number'.
 */
export function sum(list: Array<number>): number {
    let result = 0;
    for (const val of list) {
        result += val;
    }
    return result;
}

Unit testing vs. Integration testing

  • 单元测试（unit test）定义为对单个模块隔离进行测试，上文所述皆属此部
  • 融合测试（integration test）定义为同时对多个模块一起进行的测试

融合测试是必要的，它能按照各模块之间的依赖关系对其进行校验。也正因此，对模块集合的融合测试必须在被其依赖的各个模块经单元测试通过后进行，以符合 立即出错 的原则

Iterative test-first programming

Iterate on a target module from scratch:

1. Update specification for the module.

2. Update tests that exercise the spec. If any problem is found, goto step 1.

3. Update implementation until it can pass all tests. If any problem is found, goto step 1.

After the basic version is constructed, iterate the module while using it:

4. If any optimizing changes of implementation is done, goto step 3.

5. If any bug of this module is found, add it to the test set, then goto step 1.
